# SPEC.md — Cortex

> The continuity protocol for Claude surfaces.

Build stage: Stage 7
Drift status: CURRENT
Last updated: 2026-03-15

---

## Overview

Cortex defines how context flows between Claude surfaces — Chat (web/mobile), Code (terminal), and future surfaces (API agents, desktop, extensions). The protocol is file-based, human-readable, and works with today's Claude infrastructure.

The core abstraction: every Claude surface both **produces** and **consumes** context objects. Cortex defines the schema for those objects, the contracts for each surface, and the transport layer that moves them.

---

## Context Schema

A context object is a structured record of something Claude learned or produced on one surface that would be valuable on another. Every context object follows this format:

```yaml
---
id: ctx_<uuid>
type: decision | artifact | state | priority | blocker | insight
source_surface: chat | code | api | desktop
timestamp: ISO-8601
project: string | null          # null = cross-project / personal
confidence: high | medium | low # how certain is this context
ttl: persistent | session | 24h | 7d  # when does this expire
supersedes: ctx_<uuid> | null   # replaces a previous context object
tags: string[]
---

# Title

Body — human-readable description of the context.

## Structured Data (optional)

key: value pairs that surfaces can parse programmatically.
```

### Context Types

| Type | What It Captures | Example |
|------|-----------------|---------|
| `decision` | A choice made that affects future work | "Use email OTP for auth, not passwords" |
| `artifact` | Something produced — code, docs, designs | "Shipped v2 of the billing API" |
| `state` | Current status of work in progress | "Auth module 80% done, blocked on Stripe webhook" |
| `priority` | What matters most right now | "Revenue before new features this week" |
| `blocker` | Something preventing progress | "Can't deploy until DNS propagates" |
| `insight` | A realization that changes understanding | "Users abandon onboarding at step 3, not step 5" |

### Field Rules

- **`id`**: Generated by the producing surface. Format: `ctx_` + UUIDv4 (first 8 chars). Example: `ctx_a1b2c3d4`.
- **`confidence`**: Surfaces should be honest. A casual mention in Chat is `low`. An explicit decision is `high`. A commit is `high`.
- **`ttl`**: Most decisions are `persistent`. Blockers are often `24h` or `7d`. State is `session` by default (refreshed each session).
- **`supersedes`**: When a new decision replaces an old one, link them. This prevents stale context from overriding current context.
- **`project`**: Maps to a project directory or identifier. `null` means it applies everywhere (e.g., "I prefer terse responses").

## Surface Contracts

Each Claude surface has a defined contract: what it produces and what it consumes.

### Chat (Web / Mobile)

**Produces:**
- `decision` — Architectural choices, strategic direction, priority shifts
- `priority` — What the human cares about right now
- `insight` — Realizations from conversation that affect the work
- `blocker` — Frustrations, confusions, or obstacles mentioned in conversation

**Consumes:**
- `artifact` — What Code just shipped (commits, structure changes, new patterns)
- `state` — Where Code is in the build (% complete, active branch, failing tests)
- `blocker` — What Code is stuck on (so Chat can help reason through it)

**How Chat consumes:** Context objects are injected into Chat's system prompt or memory as structured summaries. Chat sees a "build state" section that updates after each Code session.

### Code (Terminal)

**Produces:**
- `artifact` — Commits, new files, architecture changes, dependency additions
- `state` — Build progress, test results, branch status, active TODOs
- `blocker` — Errors, failing tests, unresolved technical decisions

**Consumes:**
- `decision` — Strategic choices made in Chat that affect implementation
- `priority` — What to work on first, what can wait
- `insight` — User frustrations or realizations that should influence how Code approaches the work

**How Code consumes:** Context objects are written to `.claude/cortex/` as markdown files. Claude Code reads them as part of its project context (similar to how CLAUDE.md works today).

### API / Agents (Future)

**Produces:**
- `artifact` — Task completions, data transformations, report outputs
- `state` — Agent progress, queue depth, error rates

**Consumes:**
- `priority` — Which tasks matter most
- `decision` — Parameters, constraints, and boundaries set by the human

## Transport Layer

### Storage: The Cortex Store

Context objects live in a **local-first, file-based store**:

```
~/.cortex/
  contexts/                    # All context objects
    ctx_a1b2c3d4.md           # Individual context files
    ctx_e5f6g7h8.md
  index.json                   # Fast lookup index
  surfaces/                    # Per-surface sync state
    chat.json                  # Last sync timestamp, pending contexts
    code.json
  config.yaml                  # User preferences, project mappings
```

Why file-based:
1. Works offline
2. Human-inspectable (you can `cat` any context object)
3. Git-trackable (optional — version your context history)
4. No external dependencies
5. Matches Claude Code's existing `.claude/` pattern

### Sync Mechanism

**Code -> Store:** A Claude Code hook (PostToolUse or session end) extracts context objects from the session and writes them to `~/.cortex/contexts/`. The hook runs automatically — zero effort from the human.

**Store -> Code:** At session start, Claude Code reads `~/.cortex/contexts/` for objects relevant to the current project. These are injected via a CLAUDE.md include or a dedicated `.claude/cortex/` directory that Claude Code auto-reads.

**Chat -> Store:** A Claude Chat integration (browser extension or API middleware) captures decisions and priorities from conversations and writes them to the store. Initially, this is manual (the user tells Chat to "save this decision"). Later, it's automatic.

**Store -> Chat:** Chat reads from the store at conversation start. Context objects are formatted as a structured summary in the system prompt or injected via Memory.

### Sync Triggers

| Trigger | Direction | Mechanism |
|---------|-----------|-----------|
| Code session start | Store -> Code | Read `~/.cortex/contexts/`, filter by project |
| Code commit | Code -> Store | PostToolUse hook extracts artifacts + state |
| Code session end | Code -> Store | Summarize session, write state objects |
| Chat conversation start | Store -> Chat | Read store, format as context summary |
| Chat explicit save | Chat -> Store | User says "remember this" or "save this decision" |
| Chat conversation end | Chat -> Store | Extract decisions and priorities automatically |

### Conflict Resolution

When two surfaces produce conflicting context:

1. **Newer wins by default.** Timestamps determine recency.
2. **`supersedes` breaks ties.** If a context explicitly replaces another, the replacement wins.
3. **Human arbitration on demand.** The user can view conflicts via `cortex status` and resolve manually.
4. **Confidence weights.** A `high` confidence context from Code (a commit) outweighs a `low` confidence context from Chat (a passing mention).

## Privacy and Control

### Principles

- **Nothing syncs without the user knowing.** The `cortex status` command shows exactly what's in the store.
- **Project-level scoping.** Users can restrict sync to specific projects. Personal conversations stay personal.
- **Opt-out per context.** Any context object can be deleted from the store. The user owns their context.
- **No cloud by default.** Everything is local. Cloud sync is opt-in for multi-device setups.

### Commands

```bash
cortex status                  # What's in the store, what's pending sync
cortex list                    # All context objects, filterable by type/project
cortex show ctx_a1b2c3d4       # View a specific context object
cortex delete ctx_a1b2c3d4     # Remove a context object
cortex sync                    # Force sync (normally automatic)
cortex config                  # Edit preferences
cortex export                  # Export store as JSON (for backup or migration)
```

## Reference Implementation

### Phase 1: Code-side (works today)

Build a Claude Code hook + MCP server that:
1. Writes context objects to `~/.cortex/` on commit and session end
2. Reads context objects at session start and injects them into project context
3. Provides `cortex` CLI commands for inspection and management

**Stack:** TypeScript, runs as a Claude Code hook + optional MCP server.

### Phase 2: Chat-side (requires extension or API)

Build a browser extension or Anthropic API integration that:
1. Captures decisions and priorities from Chat conversations
2. Writes them to `~/.cortex/` (local) or a sync layer (cloud)
3. Reads Code-produced context and surfaces it in Chat

**Stack:** Browser extension (Chrome/Firefox) or Anthropic Memory API integration.

### Phase 3: Protocol formalization

Publish Cortex as a formal protocol spec that:
1. Any Claude surface (or third-party tool) can implement
2. Includes conformance tests
3. Becomes a community standard for Claude continuity
